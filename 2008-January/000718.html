> I don't remember exactly the docs, but there are some status-codes that
> can be returned together with the "reg" event package, which indicate a
> lot of the intended use.
>
> The RTR could indicate a permanent de-registration and the user should
> not retry to re-register - e.g. subscription terminated immediately. Or
> it could be temporary and the user should retry - for SIM card
> invalidation or when there is a doubt about a user's security measures.
> Also something interesting that can be achieved for a carrier grade
> network would be to move subscribers from an S-CSCF to another in a nice
> manner - then they all receive such a RTR which just indicates that they
> should re-register. When they do, the network would orchestrate a nice
> re-direction.
>
