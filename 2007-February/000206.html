Hi Luis, 

here some minor thoughts ...

On Tue, 2007-02-13 at 12:09 +0100, Dragos Vingarzan wrote:  
> > I'm studying your HSS and I noticed that it support release 6 and of course you have a relationship of many-to-many between the private and the public identities. The problem for me arises when you define the registration state only related with the public identity. For me, and according to the spec (I think), the registration state is related with a set of identities (i.e, one private and one public). So, when you are processing a registration you are registring a set of a private and a public rather then only the public.
> >   
> If you are referring to the implicit set, this is being working on, I
> think. On the S-CSCF this should already be working.

Correct, we are looking into this.
> >  
> > "For Shared Public User Identities, the S-CSCF shall initiate this procedure towards the HSS for each Private User Identity undergoing a Registration or Deregistration related to the Shared Public User Identity. " - 29.228
> >   
> I think this is a little out of context. This implies that once a SIM
> card registers, the S-CSCF should automatically register the other? I
> don't think so

Me neither. Do you need such a feature ?
  
> > I understand shared public identities a set of public identities belonging to the same private. So you can register, for example, the public identity sip:bob at ptinovacao.pt with the private bob1 at ptinovacao.pt but you can also register the same public with a different private (bob2 at ptinovacao.pt). You have to consider two different registrations.
> >   
> yes. This is new to R6 from R5 - a public associated to many privates.
> This is a hard topic, but should be manageable.

A deficit that we carried over with us and never upgraded so far. We are
currently fixing this along with some other issues.
> >  
> > This is the case of Deregistration:
> > "If the Public User Identity is currently registered with more than one Private User Identity, the HSS shall keep the registration state of the Public User Identity as Registered and retain the S-CSCF name associated with the Public User Identity."
> >   
> This is a little difficult. The S-CSCF deals with contacts. And now, it
> only sends a SAR for deregistration when all contacts have been deleted.
> Sending one earlier might confuse a HSS that is not implementing what
> your are saying above and leave the user in an inconsistent state
> between the S-CSCF and HSS.
> >  
> > Can you handle this case, according with your database structure and logic?
> >   
> If not, we should change it :).

We are :) The downside will be some changes in the database setup.

- Peter
